# stock_system_good_us_v2 — 미국 주식 +10% 급등 후보 스캔

v1(stock_system_good_us) 전체 코드 리뷰에서 발견된 문제를 모두 반영해
**처음부터 다시 작성한** 시스템입니다.

## 예측 목표 (명시적 정의)

> **향후 10거래일 내에 고가 기준 오늘 종가 대비 +10% 이상 도달할 확률**

v1은 "하루 +4% 상승"의 전조 스테이지(D-5~D-1)를 예측했는데, 이는
목표(+10% 급등)와 다른 대상이었습니다. v2는 목표를 그대로 라벨로 씁니다.

## 사용법

```bash
pip install -r requirements.txt

# 1) 모델 학습 (최초 1회 + 주 1회 갱신 권장, 10~20분 소요)
python3 surge_model.py train

# 2) 매일 장 마감 후 스캔
python3 daily_scan.py ALL              # 최신 종가 기준
python3 daily_scan.py ALL 2026-07-17   # 특정일 종가 기준

# 3) 요약 이메일 (최신 스캔 JSON 파싱 — 하드코딩 없음)
python3 summary_email.py
```

이메일: `email_sender.py` 최초 실행 시 생성되는 `email_config.json`에
Gmail 앱 비밀번호를 넣고 `"enabled": true` 로 변경.

## v1 리뷰 지적사항 → v2 반영 내역

| # | v1 문제 | v2 해결 |
|---|---------|---------|
| 1 | `surge_predictor` 모듈 부재로 "급등임박 45%" 점수가 항상 0 (조용히 삼켜짐) | 외부 의존 전부 제거. 확률 모델(`surge_model.py`)이 리포 내부에 있고, 모델이 없으면 **리포트에 "모델 미학습" 명시** — 조용한 0 혼합 금지 |
| 2 | 모델 피처 7개(RangeRel, OBVSlope5, StochK14 등)가 계산되지 않아 항상 NaN | `indicators.py`가 `MODEL_FEATURES` 전부를 실제 계산. `validate_features()`가 학습·예측 시 누락/전체-NaN을 **즉시 에러**로 차단 |
| 3 | `stage_tracker` 부재로 정밀도 필터·학습 루프 미작동, 모델 pkl은 gitignore | `tracker.py` 내장: 예측 기록 → 기한 도래 시 실측(+10% 도달 여부) 평가 → 실현 정밀도 기반 게이트 갱신. `surge_model.pkl`, `predictions/`, `tracker_stats.json` 모두 **git 커밋 대상** |
| 4 | 급등 정의 4% ≠ 목표 10%, 고정 임계값이 소형주 라벨을 노이즈화 | 라벨 = "10거래일 내 +10% 도달" 이진 분류. 스캔 대상과 동일한 가격/유동성 필터를 학습 데이터에도 적용 |
| 5 | VCP/OBV_Diverge가 마지막 행에만 계산 → 학습-실전 불일치 | 전 구간 **벡터화** 계산 — 학습과 실전이 같은 값을 봄 |
| 6 | 생존 편향: 현재 생존 종목으로 과거 백테스트 → 적중률 부풀림 | 완전 해결은 불가(시점 고정 유니버스 데이터 필요). 대신 백테스트 출력에 **경고 명시** + 실측 추적(`tracker`)을 신뢰 기준으로 삼음. SGEN(상폐)·FISV(→FI) 정리 |
| 7 | 수축(VCP/NR7)과 확대(ATR/BB) 모순 신호에 동시 가점 | 셋업 섹션에서 **둘 중 높은 경로 하나만** 반영 |
| 8 | 섹션 상한 미준수(명목 20점 섹션이 실제 79점) → 상위권 점수 포화 | 모든 섹션에 `min()` **상한 강제** (추세20/수급30/OB20/셋업20/스퀴즈10) |
| 9 | 이미 급등한 종목 추격 (5일 +30%에 겨우 -10점) | 5일 +15% / 10일 +25% 시 각 -20점. 엘리트픽에서 과열 종목 **제외** |
| 10 | 언더샘플링+balanced 학습의 미보정 확률을 "신뢰도 90%"로 표시 | **Isotonic 확률 보정** + 검증셋에서 목표 정밀도 만족 임계값 선택·저장. 리포트 확률은 보정값 |
| 11 | 요약 이메일에 특정일 결과 하드코딩 (한 달 전 추천을 재발송) | `summary_email.py`는 최신 `reports/*_scan.json` 파싱 — 하드코딩 없음 |
| 12 | 데이터 실패 조용히 삼킴 (rate limit 걸린 날 점수가 몰래 달라짐) | `FetchLog`가 실패 종목을 수집해 **리포트에 출력**. info 실패 시 0 조작 대신 `data_ok=False`로 해당 항목 제외 |
| 13 | 태그 "20>50>200" vs 실제 MA20/60/120 불일치 | MA 체계를 US 관행 **20/50/200으로 통일** (계산=표기) |
| 14 | 스캔이 코드를 자동 commit/push | **제거** — 커밋은 사람이 검토 후 수행 |
| 15 | 잔버그: `pvm and pvm >= 0`(0.0이 ❌), fast_info.company 오용, OB 위쪽인데 "눌림목 대기" 표기, 종목마다 SPY 재수집 | 전부 수정 (`is not None` 비교, info.longName 사용, "돌파 확인 후 진입" 표기, SPY 스캔당 1회 수집) |
| 16 | 자체 분석상 무의미한 RSI 35~55 가점(+6) | 제거. 과매수 페널티만 유지 |
| 17 | 워크포워드에 라벨 미래구간 누출 가능성 | 학습/테스트 사이 HORIZON(10일) **퍼지 갭** 적용, 라벨 없는 마지막 10일 학습 제외 |

## 파일 구성

```
config.py        목표(+10%/10일)·필터·가중치 설정
universe.py      유니버스 (생존 편향 경고 포함)
data_fetcher.py  yfinance 수집 + FetchLog (실패 명시)
indicators.py    전 지표 벡터화 + MODEL_FEATURES + validate_features
order_block.py   오더블록 + 진입/목표/손절 계산
scorer.py        기술 점수 (섹션 상한 강제, 과열 페널티)
surge_model.py   +10%/10일 이진 분류 (보정 확률, 퍼지 갭 워크포워드)
tracker.py       예측 기록 → 실측 평가 → 실현 정밀도 게이트
daily_scan.py    오케스트레이션 (2단계 스캔, 리포트, 이메일)
summary_email.py 최신 스캔 JSON 기반 요약 (하드코딩 없음)
email_sender.py  Gmail SMTP
```

## 한계 (정직한 고지)

- **생존 편향**: 유니버스가 현재 생존 종목 목록이므로 백테스트 수치는
  낙관적입니다. 시스템의 진짜 성적은 운영하며 쌓이는 `tracker_stats.json`
  (실측 적중률)으로 판단하세요.
- **공매도 데이터**: yfinance `shortPercentOfFloat`는 격주 갱신이라 최대
  2주 지연입니다 (리포트에 표기됨).
- **base rate 주의**: 변동성 장세에서는 아무 종목이나 골라도 10일 내 +10%에
  자주 도달합니다. 백테스트 출력의 base rate 대비 lift로 판단하세요.
- 이 시스템은 참고용이며 투자 판단과 책임은 본인에게 있습니다.
